    Minimizing LaTeX Docker Image with strace and ldd

    When building automated systems that generate PDFs from LaTeX templates, one often runs into the “huge image” problem. A full TeX Live installation can easily exceed 5GB, and even a “recommended” set of packages often lands around 2-3GB.

    In this post, I’ll document how I managed to shrink a LaTeX Docker image from 2.65GB down to just 33.6MB—a reduction of over 98%.

    The Challenge

    The goal was to create a lightweight image for an OpenWRT router that can compile a specific invoice template. The template uses several packages like geometry, opensans, eurosym, tikz, and hyperref.

    The Strategy: Dynamic Dependency Analysis

    Instead of guessing which LaTeX packages are needed, we use a more scientific approach:

    1. Build a full “builder” image with all potential dependencies.
    2. Run the actual compilation workload under strace.
    3. Capture every file the process successfully accesses.
    4. Resolve library dependencies using ldd.
    5. Create a “minimal” final image containing only those files.

    Step 1: The Builder and the Analysis Script

    I created a shell script analyze_deps.sh that automates the discovery:

    #!/bin/sh
# Runs a command and finds all file/library dependencies.

LOG_FILE="trace.log"
OUT_FILE="needed_files.txt"
CMD="$@"

echo "Step 1: Running workload under strace..."
strace -f -e trace=openat,stat,lstat,access -o "$LOG_FILE" $CMD

echo "Step 2: Extracting paths from strace log..."
grep -v "ENOENT" "$LOG_FILE" | grep -oE '"/[^"]+"' | sed 's/"//g' | sort | uniq > raw_paths.txt

echo "Step 3: Resolving symlinks and library dependencies..."
touch deps_all.txt
while read -r p; do
    if [ -e "$p" ]; then
        echo "$p" >> deps_all.txt
        target=$(readlink -f "$p")
        [ "$target" != "$p" ] && echo "$target" >> deps_all.txt
        if ldd "$p" > /dev/null 2>&1; then
            ldd "$p" | awk '{for(i=1;i<=NF;i++) if($i ~ /^\//) print $i}' >> deps_all.txt
        fi
    fi
done < raw_paths.txt

echo "Step 4: Finalizing list..."
sort deps_all.txt | uniq | grep -vE "^(/proc|/dev|/sys|/tmp|/work|/etc/passwd|/etc/group)" > "$OUT_FILE"

    Step 2: Multi-stage Dockerfile

    The Dockerfile leverages the builder stage to generate the list of needed files and then copies them to a clean alpine base.

    # Stage 1: Builder
FROM alpine:latest AS builder
RUN apk add --no-cache texlive texmf-dist-latexextra texmf-dist-fontsextra texmf-dist-fontsrecommended strace
WORKDIR /work
COPY analyze_deps.sh /work/analyze_deps.sh
COPY input.tex /work/input.tex

# Generate the list of needed files
RUN /bin/sh /work/analyze_deps.sh pdflatex /work/input.tex

# Prepare the minimal filesystem
RUN mkdir /dist && \
    while read p; do \
        if [ -e "$p" ]; then \
            cp --parents "$p" /dist/ ; \
            target=$(readlink -f "$p") ; \
            [ "$target" != "$p" ] && cp --parents "$target" /dist/ ; \
        fi ; \
    done < /work/needed_files.txt && \
    cp --parents /bin/sh /dist/ && \
    cp --parents /lib/ld-musl-x86_64.so.1 /dist/

# Stage 2: Final minimal image
FROM alpine:latest
COPY --from=builder /dist /
WORKDIR /work
ENTRYPOINT ["pdflatex"]

    Results

    The difference is staggering:

    • Original Image: 2.65GB
    • Minimal Image: 33.6MB

    The final image contains only the pdflatex binary, its required shared libraries, the specific .fmt files, and just the font and style files actually used by the template. This makes it perfect for deployment on resource-constrained devices like routers or as a fast-loading part of a CI/CD pipeline.
